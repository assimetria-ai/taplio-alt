## Issue Identified

The task description indicated that `webhook-handler.js:18` needed SubscriptionRepo and UserRepo. Upon investigation, the webhook handling code was found in `server/src/api/@system/stripe/index.js` instead of a separate `webhook-handler.js` file.

### The Problem

While the webhook handlers were mostly implemented, **`stripe_customer_id` was not being persisted** in subscription records during webhook events. This caused a critical issue:

**Symptom:**
```javascript
// POST /api/stripe/create-portal-session
if (!subscription?.stripe_customer_id) {
  return res.status(404).json({ message: 'No active subscription found' })
}
```

The billing portal endpoint checks for `stripe_customer_id` in the subscription record, but the webhook handlers were not saving this field, causing portal session creation to fail.

### Missing Data Flow

**Before Fix:**
```
Checkout webhook → Create subscription record → ❌ No stripe_customer_id saved
User tries to open billing portal → ❌ 404: No active subscription found
```

**Root Cause:**
Four webhook handlers were creating/updating subscription records without including `stripe_customer_id`:
1. `checkout.session.completed`
2. `customer.subscription.created`
3. `customer.subscription.updated`
4. `invoice.payment_succeeded`

---

## The Fix

Added `stripe_customer_id` to all webhook handlers that create or update subscription records.

**File:** `product-template/server/src/api/@system/stripe/index.js`

### 1. checkout.session.completed

**Before:**
```javascript
await SubscriptionRepo.upsertByStripeSubscriptionId({
  user_id: Number(userId),
  stripe_subscription_id: stripeSubscription.id,

  status: stripeSubscription.status,
  current_period_start: new Date(stripeSubscription.current_period_start * 1000),
```

## Impact on Existing Subscriptions

### Migration Consideration

**Issue:** Existing subscriptions created before this fix may have `NULL` for `stripe_customer_id`.

**Solution Options:**

1. **Automatic backfill on next webhook:**
   - Next `customer.subscription.updated` or `invoice.payment_succeeded` will populate the field
   - No manual migration needed

2. **Manual backfill script (if immediate fix needed):**
```javascript
// scripts/backfill-stripe-customer-ids.js
const SubscriptionRepo = require('./src/db/repos/@system/SubscriptionRepo')
const stripe = require('./src/lib/@system/Stripe')

async function backfill() {
  const subs = await SubscriptionRepo.findAll()
  for (const sub of subs) {
    if (!sub.stripe_customer_id && sub.stripe_subscription_id) {
      const stripeSub = await stripe.subscriptions.retrieve(sub.stripe_subscription_id)
      await SubscriptionRepo.update(sub.id, {
        stripe_customer_id: stripeSub.customer
      })
      console.log(`Backfilled customer ID for subscription ${sub.id}`)
    }
  }
}

backfill().then(() => console.log('Done')).catch(console.error)
```
