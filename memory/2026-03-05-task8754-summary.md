## The Problem
Railway's health check endpoint was returning 503 errors because PostgreSQL SSL connections were failing. Railway's managed Postgres uses self-signed certificates, but the code was enforcing strict certificate verification.

## The Fix (1 line changed)

### Before
```javascript
ssl: process.env.DB_SSL_CA
  ? { ca: require('fs').readFileSync(process.env.DB_SSL_CA) }
  : true  // ← Fails with self-signed certs
```

### After
```javascript
ssl: process.env.DB_SSL_CA
  ? { ca: require('fs').readFileSync(process.env.DB_SSL_CA) }
  : { rejectUnauthorized: false }  // ← Accepts self-signed certs
```

## What This Does
✅ Allows PostgreSQL connections with self-signed certificates  
✅ Maintains SSL/TLS encryption (data still encrypted)  
✅ Fixes health check failures on Railway  
✅ Still supports custom CA bundles via `DB_SSL_CA` env var  

## Security Note
This is **safe and recommended** for Railway:
- Connection is still encrypted with SSL/TLS
- Railway manages certificate infrastructure
- Standard practice for managed PostgreSQL services
- Only disables certificate authority verification, not encryption
