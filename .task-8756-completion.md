## Issue

Template QA detected that @system code was importing from @custom, which violates the architecture principle that @system should be self-contained and @custom should build on top of @system (not the other way around).

### Problematic Code

**File:** `server/src/scheduler/tasks/@system/scheduler.js`

```javascript
async init() {
  try {
    const initCustom = require('../@custom/init')
    initCustom(this)
    logger.info('[scheduler] custom tasks initialised')
  } catch (err) {
    logger.warn({ err }, '[scheduler] no custom init found or init failed — skipping')
  }
}
```

This method in the @system scheduler was importing from `@custom/init`, creating a circular dependency where system-level code depends on custom-level code.

---

## Solution

Moved the custom task initialization logic from @system to the application bootstrap layer (`src/index.js`), which is the appropriate place for wiring together @system and @custom modules.

### Changes Made

#### 1. Removed `init()` method from @system/scheduler.js

Deleted the `init()` method entirely from the Scheduler class, keeping @system free from @custom dependencies.

#### 2. Updated application bootstrap (index.js)

Replaced:
```javascript
await scheduler.init()
```

With:
```javascript
// Initialize custom tasks (application layer imports @custom, not @system)
try {
  const initCustomTasks = require('./scheduler/tasks/@custom/init')
  initCustomTasks(scheduler)
  logger.info('custom tasks initialised')
} catch (err) {
  logger.warn({ err }, 'no custom task init found or init failed — skipping')
}
```

#### 3. Updated @custom/init.js comments

Changed the comment to reflect that it's called from `src/index.js` rather than `scheduler.init()`.
