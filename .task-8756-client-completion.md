## Issue

Template QA detected that @system code was importing from @custom on the **client-side**, violating the architecture principle that @system should be self-contained and @custom should build on top of @system (not the other way around).

The server-side was already fixed in previous commits (ab28170, 9db3ed5, 53fee00, b969e08), but client-side violations remained.

### Problematic Code

**File 1:** `client/src/app/pages/static/@system/LandingPage.jsx`
```javascript
const FAQ = lazy(() =>
  import('../../../components/@custom/FAQ')
    .then((m) => ({ default: m.FAQ }))
    .catch(() => ({ default: () => null }))
)
const HeroSection = lazy(() =>
  import('../../../components/@custom/HeroSection/HeroSection')
    .then((m) => ({ default: m.HeroSection }))
    .catch(() => ({ default: () => null }))
)
```

**File 2:** `client/src/app/pages/static/@system/RegisterPage.jsx`
```javascript
const PasswordStrengthIndicator = lazy(() =>
  import('../../../components/@custom/PasswordStrengthIndicator')
    .then((m) => ({ default: m.PasswordStrengthIndicator }))
    .catch(() => ({ default: () => null }))
)
```

**File 3:** `client/src/app/routes/@system/AppRoutes.jsx`
```javascript
const BlogAdminPage = lazy(() =>
  import('../../pages/app/@custom/BlogAdminPage').then((m) => ({ default: m.BlogAdminPage }))
)

// ...and later:
useEffect(() => {
  import('../@custom')
    .then((mod) => setCustomRoutes(mod.customRoutes ?? []))
    .catch(() => setCustomRoutes([]))
}, [])
```

---

## Solution

Removed all @custom imports from @system files. @system components are now completely self-contained. Products should override pages in @custom or create wrapper components instead of having @system depend on @custom.

### Changes Made

#### 1. LandingPage.jsx

**Removed:**
- Lazy imports for `FAQ` and `HeroSection` from @custom
- Suspense wrappers for these components

**Added:**
- Simple, self-contained hero section using only @system components
- Comment instructing products to create `@custom/LandingPage.jsx` if they want custom sections

**Result:** @system LandingPage is now fully functional without any @custom dependencies.

#### 2. RegisterPage.jsx

**Removed:**
- Lazy import for `PasswordStrengthIndicator` from @custom
- Suspense wrapper and component usage

**Result:** @system RegisterPage works standalone. Products can create `@custom/RegisterPage.jsx` to add password strength indication.

#### 3. AppRoutes.jsx

**Removed:**
- Import of `BlogAdminPage` from @custom
- `useState` and `useEffect` for loading custom routes
- Route definition for `/app/admin/blog`
- `{customRoutes}` placeholder in route tree

**Result:** @system AppRoutes defines only template routes. Products should create `@custom/AppRoutes.jsx` to add custom routes.

## Migration Guide for Products Using This Template

If a product was built from the template before this fix, they may need to:

1. **For custom landing sections:** Create `@custom/LandingPage.jsx` that imports and composes with `@system/LandingPage`
2. **For password strength:** Create `@custom/RegisterPage.jsx` that wraps `@system/RegisterPage` and adds the indicator
3. **For custom routes:** Create `@custom/AppRoutes.jsx` that imports `@system/AppRoutes` and adds custom route definitions

Alternatively, products can keep using their existing @custom components by importing them at the application layer (e.g., in `App.jsx` or route definitions outside of @system).
