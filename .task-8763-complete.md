## Issue Identified

The automated template QA detected that `product-template/server/src/app.js` was missing a `/health` endpoint. While it had `/healthz` (Kubernetes convention), it lacked the standard REST `/health` endpoint that many monitoring tools and orchestrators expect.

### The Problem

**Before:**
```javascript
// Only /healthz endpoint
app.get('/healthz', (_req, res) => res.status(200).json({ status: 'ok' }))
```

**Compatibility Issues:**
- ❌ Many monitoring tools expect `/health` by convention
- ❌ Some orchestrators default to `/health` for readiness probes
- ❌ REST APIs typically use `/health` not `/healthz`
- ❌ Only supporting Kubernetes convention limits deployment flexibility

**Examples of tools that expect `/health`:**
- AWS Application Load Balancer (default health check path)
- Azure App Service (default health check)
- Docker Compose / Swarm health checks
- Consul service mesh
- Nagios / Prometheus (common convention)
- Many APM tools (New Relic, Datadog, etc.)

---

## The Fix

Added `/health` endpoint alongside `/healthz` for maximum compatibility.

**File:** `product-template/server/src/app.js`

**After:**
```javascript
// Health check endpoints registered before all middleware (including CORS) so that
// infrastructure health probes with no Origin header reach them without triggering
// CORS rejection. These are the only paths permitted to bypass CORS in production.
//
// Both /health and /healthz are provided for compatibility:
// - /health: Standard REST convention
// - /healthz: Kubernetes/GKE convention
app.get('/health', (_req, res) => res.status(200).json({ status: 'ok' }))
app.get('/healthz', (_req, res) => res.status(200).json({ status: 'ok' }))
```

## Endpoint Comparison

The template now provides **three health endpoints** for different use cases:

| Endpoint | Purpose | Middleware | DB Check | Use Case |
|----------|---------|------------|----------|----------|
| **`/health`** | Infrastructure probe | ❌ No | ❌ No | Standard REST health checks |
| **`/healthz`** | Infrastructure probe | ❌ No | ❌ No | Kubernetes/GKE probes |
| **`/api/health`** | Application health | ✅ Yes | ✅ Yes | Authenticated deep health check |

**Why three endpoints?**
1. **`/health` & `/healthz`** - Fast, no-dependency checks for load balancers
2. **`/api/health`** - Comprehensive check including database connectivity

---

## Implementation Details

### Endpoint Characteristics

**Both `/health` and `/healthz`:**
- ✅ Registered before all middleware
- ✅ Bypass CORS restrictions
- ✅ No authentication required
- ✅ No database check (fast response)
- ✅ Always return 200 OK if server is running
- ✅ Return JSON: `{ "status": "ok" }`

**Why no database check?**
- Infrastructure probes need fast, reliable responses
- DB connection issues shouldn't mark the server as "down"
- Use `/api/health` for deep health checks with DB verification

### Response Format

```json
{
  "status": "ok"
}
```

**Status:** 200 OK (server is running)

## Security Considerations

### Why Bypass CORS?

Health check endpoints bypass CORS because:
1. **Infrastructure tools don't send Origin headers** - Load balancers, Kubernetes, etc.
2. **No sensitive data exposed** - Just `{ "status": "ok" }`
3. **Public information** - Server being "up" is not secret
4. **Operational requirement** - Health checks must always work

### What's NOT Exposed

❌ Database connection details  
❌ Redis connectivity  
❌ Service versions  
❌ Infrastructure topology  
❌ Internal metrics  

**For detailed health info:** Use authenticated `/api/health` endpoint
